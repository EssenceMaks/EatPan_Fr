# EatPan — Figma Synchronization Rules

Для интеграции с макетами мы используем Figma MCP Server. Все обновления UI должны сверяться с актуальным дизайном.

## Ссылки и ключи
- **Figma URL:** `https://www.figma.com/design/lWzzBp0TFIsbeypUlRMSxu/EatPan?node-id=0-1&t=kmydVw7qRqV57D7s-1`
- **File Key:** `lWzzBp0TFIsbeypUlRMSxu`

## Workflow (Порядок действий агента)
1. **Проверка макета:** Перед созданием новых модулей, компонентов или изменением стилей, агент обязан считать структуру/стили из Figma (используя инструменты MCP сервера для Figma, передав File Key).
2. **Дизайн-токены:** Все цвета, шрифты и отступы из Figma должны маппиться на CSS-переменные в `src/styles/global.css` (внутри `:root`). Если в Figma появился новый цвет или шрифт — его нужно добавить в `:root`.
3. **Реализация в коде:** Переносить дизайн из Figma строго в парадигме проекта: Vanilla JS (компоненты на базе `src/core/Component.js`), БЭМ-нейминг классов, без Tailwind и сторонних библиотек. Для иконок использовать Lucide.
4. **Синхронизация:** Если разработчики (команда) меняют макет в Figma, агент должен уметь получать эти обновления по запросу пользователя и вносить соответствующие правки в HTML/CSS/JS проекта EatPan_Fr.

## Разрешение конфликтов и логгирование (Перепроверка)
При сверке актуального дизайна агент должен учитывать, что другой разработчик мог внести локальные улучшения напрямую в код (например, настроил отступы в DevTools, поменял иконку "птички"), которые еще не отражены в Figma.

- **ЗАПРЕЩАЕТСЯ** слепо перезаписывать локальный код данными из Figma, если обнаружены расхождения.
- **ОБЯЗАТЕЛЬНАЯ ФИКСАЦИЯ:** При обнаружении отличий фронтенд-элементов (компонентов, классов, стилей) агент **обязан** занести эти расхождения в отдельный MD-файл: `docs/changelogs/figma_validation.md`. Там нужно кратко и грамотно указать названия элементов и участки кода, где они находятся.
- **ОБЯЗАТЕЛЬНАЯ КОММУНИКАЦИЯ:** После записи в файл валидации, агент должен явно попросить пользователя зайти в этот файл, посмотреть список и сделать выбор (заверить правки):
  1. *Вариант А (Figma -> Код):* Взять из Figma и внести правки в проект.
  2. *Вариант Б (Код -> Figma):* Перенести локальные правки из кода обратно в Figma (сохранить код).
- **ИСТОРИЧЕСКОЕ ЛОГИРОВАНИЕ:** Как только выбор сделан и правки применены (синхронизация завершена), агент **обязан** внести запись об успешном слиянии в файл истории `docs/changelogs/figma_changelog.md`. Это обеспечит сохранность истории версий и возможность отката при ошибках дизайна.
  
Главный ориентир: агент ничего не удаляет без явного подтверждения пользователя, всегда предлагая выбор при нестыковке версий.
