# Пошаговое руководство (Workflow): Daily Tracker

Для того чтобы гарантировать стабильную и безошибочную реализацию, RPG Daily Tracker ОБЯЗАН собираться строго в указанной последовательности. Не пытайтесь применять стили или сложные взаимодействия до тех пор, пока не будет реализована базовая модель данных и DOM.

## Фаза 1: Фундаментальный DOM и Базовые стили
1. **Инициализация Структуры Документа**:
   - Создайте корневые CSS-переменные (variables) для цветов, рамок и отступов в соответствии с документом `Rules`.
   - Постройте статичную строку заголовка:
     - Текст "Расписание на день".
     - Кнопка 🎨 для рандомизации цвета задач.
2. **Установка Генератора Сетки**:
   - Напишите цикл `for (let i = 0; i <= 23; i++)` для формирования 24-х часовых строк (`time_list_current_line`).
   - Добавьте контейнер-обертку `tasks-area` внутрь каждой часовой строки.

## Фаза 2: Строительный слой Базовых Задач
1. **Генерация Обёртки Ввода (Input Wrapper)**:
   - Внутри каждой пустой `tasksArea` инстанциируйте DOM-узел `.task-input-wrapper`, содержащий ровно 6 `.time_cells_cube` (кубиков) и поле ввода текста.
2. **Сборка функции `createTaskItem`**:
   - Реализуйте логику по созданию полноценной `.task-item`, включающей текстовый элемент заголовка, указатель длительности времени, иконки настроек (шестеренка/корзина/архив) и те самые 6 кубиков для временной шкалы. 
   - Завяжите функцию подбора цвета `getRandomColor()` на выдачу случайного валидного цвета из заготовленной RPG HEX-палитры.
3. **Синхронизация Модели Данных**:
   - Свяжите создание задачи с состоянием в виде объекта `tasksData[taskId] = { ... }`.
   - Заставьте все обновления проходить через централизованную функцию `renderCells()`. (НИКОГДА не изменяйте цвета фонов вручную кубикам вне этого цикла).

## Фаза 3: Движок Взаимодействий (Глобальный Микшер)
1. **Реализация `renderCells`**:
   - Выполните 144 индексных вычисления, чтобы создать глобальный 2D массив разметки (`globalAssignments`).
   - Итерируйтесь по строкам DOM'а.
   - Заполните строки DOM атрибутами `data-tasks` на основании свойств массива.
2. **Реализация функции `applyColorsToCube`**:
   - Внедрите конвертер `hexToRGBA`.
   - Корректно рендерьте многослойные `linear-gradient` строки (сброс нераспределённого кубика должен выполнять обнуление фона `cell.style.background = ''`).

## Фаза 4: Делегирование событий и Перетаскивание (Drag Select)
1. **Механика формирования интервалов (Span Drag)**:
   - Добавьте CSS класс `.hover-select`.
   - Добавьте флаг состояния `isDragging` в глобальный трекинг задач.
   - Перехватывайте первый клик по пустому кубику для начала "выбора интервала", слушайте движение курсора `pointermove` для визуализации (добавления класса кубикам по ходу мыши) и фиксируйте отжатие мыши (второй клик).
2. **Наполнение конфликтных массивов**:
   - В обработчике кликов парсируйте список ID: `e.target.dataset.tasks`.
   - Если полученная длина массива `length > 1`, меняйте рендеринг поповера на функцию `openPopoverConflict` вместо стандартной.

## Фаза 5: Абсолютное плавающее меню (Popover & Sidebar)
1. **Движок позиционирования**:
   - Инициируйте настройку DOM узла `#task-popover`.
   - Инжектируйте экранные координаты: `rect.top + window.scrollY - 10`.
2. **Боковая панель разрешения конфликтов**:
   - Внедрите разделяющий фрагмент CSS Grid (`140px 260px`).
   - Динамически наполняйте колонку сайдбара в зависимости от пересекающихся задач.
3. **Верификация и зачистка**:
   - Добавьте глобальный листенер `document.addEventListener('click')`, который будет скрывать всплывающее меню, если клик выпадает за пределы узлов `.time_cells_cube`, `.task-item` или границ самого `#task-popover`.
